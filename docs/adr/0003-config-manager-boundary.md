# WP MCP Manager は config マネージャーに徹し、自社 WordPress プラグインを持たない

本アプリの責務を「Claude Desktop の `claude_desktop_config.json` を管理し、エコシステムが提供する MCP/認証機構を**消費する**こと」に限定する。WordPress サーバー側のコンポーネント（custom transport、OAuth 認可サーバー、トークン交換エンドポイント等）を自社で実装・配布・保守することはしない。

## 背景

接続中の平文露出を消すには、理想的には「アプリパスを safeStorage に保持し、config には短命トークンだけを書く」アーキテクチャが望ましい。これを公式 mcp-adapter 単体で実現する手段は無く、実現するには (a) 第三者 JWT プラグイン依存、または (b) mcp-adapter の custom transport / permission callback を使った自社コンパニオンプラグインの自作が必要になる。

custom transport の permission callback はトークンの**検証**は差し込めるが、OAuth 2.1 認可サーバー一式（認可/トークンエンドポイント、PKCE、Dynamic Client Registration、リフレッシュ）は範囲外で、自作はセキュリティ事故の温床かつ大きな保守責任を伴う。

## 決定の結果としての認証方針

- **Phase 1 既定 = Application Password**（公式 mcp-adapter の標準構成で確実に動く唯一の方式）。
- **JWT = 上級オプション**（旧/第三者プラグインが入った環境向け。Phase 2 以降で検出時のみ有効化を検討）。
- **OAuth 2.1 = Phase 3**（エコシステム側の対応が整い次第。本アプリは認可サーバーを自作しない）。
- 平文露出は接続トグルによる「使うときだけ書き出す」運用で**時間的に**最小化する（消去はしない）。

## 却下した代替案

- **極小コンパニオンプラグイン（アプリパス→短命トークン交換の1エンドポイントのみ）**: 理想形を第三者依存なしで実現できるが、自社プラグインの配布・GPL・保守が発生し「依存は最小限」の方針と衝突するため Phase 1 では採らない。将来、平文露出の解消が強く求められた場合の検討カードとして残す。
- **フル OAuth 2.1 の自作**: 労力とセキュリティリスクが見合わず却下。
